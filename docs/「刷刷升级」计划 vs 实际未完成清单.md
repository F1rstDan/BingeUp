## 一、结论

当前项目已经完成了可运行的核心纵向闭环：WXT/React/TypeScript/MV3 工程、Bilibili/YouTube/通用/基础网页适配器、视频暂停与恢复、Shadow DOM 学习遮罩、全局冷却、新词与复习、连续学习、FSRS、本地存储、Popup、设置页和统计页均已有实现及自动化测试。

但项目尚不能依据计划书和 issue #1 判定为“可交给少量真实用户验证核心假设的完整 Beta”。剩余工作不是单纯 UI 润色，而是集中在以下八条产品链路：

1. 安装引导的可选性质与 Popup 状态尚未保持一致；
2. 词库仍是 12 词示例，学习目标和水平设置不影响出词；
3. 长视频定时学习只有设置，没有触发行为；
4. Popup 与设置页有多项控制不可达或只读；
5. 导出、导入、清除及多标签数据写入仍有数据完整性风险；
6. 评分纠正、答案修改次数和“近期答错”存在断链或语义错误；
7. Beta 核心观察指标不足以可靠验证产品假设；
8. 主视频识别、触发去重、自定义网站动态识别和真实浏览器验收仍有缺口。

因此，建议保持 issue #1 打开，把下述 P0 项完成并通过真实 Chrome/Edge 验收后，再宣告 Beta v0.1 完成。

## 二、基线冲突与本次采用的假设

### 2.1 规格优先级

本次按以下优先级判定差距：

1. `CONTEXT.md` 与 ADR；
2. issue #1；
3. 两份较早的计划文档。

例如，旧执行计划要求单题“提交”后立即关闭遮罩并显示角落反馈；当前领域文档和 issue #1 已把“提交”定义为先进入答题反馈，再由“明白了”等终态操作返回视频。本清单不把缺少旧式反馈 Toast 判为缺陷。

### 2.2 已确认的默认启用与权限决定

2026-07-13 已确认：Bilibili 与 YouTube 是默认支持网站，安装后以默认配置启用并取得运行所需的主机权限；安装引导只用于让用户关闭或重新开启这些网站，不是正式学习的启动门禁。用户关闭或跳过引导时保留默认配置，正式学习照常启动。用户主动添加的其他网站仍使用按需可选权限。该决定记录在 `CONTEXT.md` 与 ADR-0002，并覆盖 issue #1/#9 中“官方站点仅在引导选择后请求权限”的旧描述。

### 2.3 未验证前提

- issue #13 的评论声称 Chrome/Edge 关键场景“基本人工验收通过”，但仓库没有逐项、带环境和版本的验收记录。本次只把它视为历史人工反馈，不视为可复现证据。
- 本次没有在真实 Bilibili、YouTube 和第三方网站中加载未打包扩展；所有站点行为结论除静态代码事实外，均标记为风险或待验收。
- 自定义网站权限模式问题依据代码与 Chrome 官方权限规则推断，尚未在真实扩展权限弹窗中复现。Chrome 文档说明运行时请求必须是清单中可选来源权限的子集，使用 `https://*/*` 时请求来源需要保持相同 scheme；当前代码请求 `*://...`，因此存在 Promise 被拒绝的高概率风险。参见 [Chrome permissions API](https://developer.chrome.com/docs/extensions/reference/api/permissions) 与 [Match patterns](https://developer.chrome.com/docs/extensions/develop/concepts/match-patterns)。

## 五、P0：Beta 完成前必须闭环

### GAP-01 安装引导的可选性质与默认启用状态尚未在所有界面保持一致

**计划/规格期望**

- Bilibili 与 YouTube 安装后以默认配置启用并可立即运行正式学习；
- 安装引导只负责让用户关闭或重新开启默认支持网站；
- 关闭或跳过引导保留默认配置，不阻止内容学习流程、Popup 或设置页；
- 用户在引导中完成选择后，所选启用状态覆盖默认值。

**实际**

- 内容脚本侧已经符合最新决定：默认站点初始 `enabled: true`，正式学习不以 `onboardingCompleted` 为门禁，manifest 也声明了两个默认支持网站的主机权限；
- 引导页已有两个默认勾选的网站开关，完成引导后会保存用户选择；直接关闭引导则不写状态，读取时继续使用默认启用值；
- Popup 状态仍把 `onboardingCompleted: false` 判为 `not-onboarding`，只显示“先完成安装引导”，隐藏正常站点状态和控制。于是同一页面可能已经运行正式学习，但 Popup 却声称必须先完成引导；
- issue #16 正在修正自定义网站的精确 HTTPS 权限申请、设置页添加和主动学习入口，该部分不在本 GAP 重复处理。

**影响**

- 用户跳过引导后，正式学习与 Popup 对产品状态给出相反结论；
- Popup 的站点控制、暂停和主动学习会被不必要地阻止；
- issue #1 仍保留旧的可选权限/引导门禁描述，若不更新会继续产生错误实现。

**完成定义**

- Popup 和设置页在未完成引导时仍读取默认站点与默认设置，提供正常状态与控制；
- 完成引导只覆盖用户明确切换的网站启用状态；关闭或跳过不写入禁用状态；
- `onboardingCompleted` 仅表示用户完成过配置，不再参与正式学习或控制界面的可用性判定；
- 覆盖“首次安装后直接访问视频”“关闭引导”“全部关闭后完成引导”“以后重新开启”的行为测试。

**已有追踪**：issue #16 正在修改 Popup 与设置页，因此本 GAP 的 ticket 应在 #16 之后完成，避免覆盖其界面改动；自定义网站加入与权限不重复建单。

### GAP-02 Beta 词库、学习目标和题目质量仍停留在示例阶段

**计划/规格期望**

- 至少提供可供真实 Beta 使用的日常高频、四级、六级词库；
- 词库有可追溯来源、许可证、离线清洗/校验流程，并按词库加载；
- 用户选择的词库和自评水平影响候选新词；
- 例句语境题不能靠词形错误排除答案。

**实际**

- 内置数据只有一个“日常高频”示例词库、12 个项目自建 CC0 单词；源码注释明确写着“后续 Issue 可替换”；
- 没有 `scripts/dictionary` 构建流水线、公开原始数据记录或压缩词库资产；`BuiltInWordBank` 把全部示例数据直接放在代码内存中，IndexedDB 的 words/decks 仓库没有成为运行时词库来源；
- `selectedDeckId` 和 `selfRatedLevel` 只被保存和展示，`LearningService` 始终使用 `getDefaultDeck()` 和全部内置单词；
- 语境题用正则删除例句中的目标词及后缀，却把 lemma 作为正确选项。例如 `abandoned` 会被替换为横线，但正确选项仍是 `abandon`，可能产生明显语法错误；
- 校验器不检查 lemma 去重、例句是否包含合法目标词形、释义同义冲突及语境题词形一致性。

**影响**

- 12 个词不足以验证一周留存、每日 5 个新词和长期复习；
- 学习目标/水平设置对用户没有实际效果；
- 题目质量会污染正确率与 FSRS 数据，削弱 Beta 结论。

**完成定义**

- 建立可重复运行的词库构建、许可证记录和质量校验流程；
- 发布至少三个首发词库，并让当前词库/水平真正限定候选新词；
- 词库按需加载，不把完整大词库常驻内容脚本；
- 语境题保存并判定正确表层词形，测试时态、单复数和派生形式；
- 以真实数据样本执行唯一答案、同义冲突、例句覆盖和离线出题验收。

### GAP-03 长视频定时学习完全没有消费端

**计划/规格期望**

- 默认同一长视频只触发一次；用户开启后，同一长视频/直播可按 10、20、30 分钟或自定义间隔产生自然触发点；
- 到时仍需尊重全局冷却、页面可见性、广告状态和当前交互状态。

**实际**

- `longVideoTimedLearningEnabled` 与 `longVideoIntervalMinutes` 只存在于类型、默认值、校验器和设置页；
- `bootstrap`、适配器和 `ContentController` 均未读取这两个字段，也没有同视频计时触发模型；
- 当前 `handledIdentities` 集合会永久阻止同一页面会话中的已处理 identity 再触发，与定时学习直接冲突。

**影响**

- 设置页向用户承诺了一个没有效果的功能；
- issue #1 用户故事 13、直播定时规则和计划 M15 未完成。

**完成定义**

- 在现有 `ContentController` 高层接缝中实现无常驻 service-worker timer 的页面内定时自然触发；
- 仅用户开启时生效，并覆盖暂停/恢复、隐藏页、广告、直播、冷却、设置热更新和清理；
- 默认同视频不重复，开启后按配置间隔重复；不把冷却结束本身误作立即弹题。

### GAP-04 Popup、设置页与运行时设置只有部分闭环

**计划/规格期望**

- Popup 显示域名、启用状态、兼容等级、覆盖方式、视频控制能力和今日摘要；
- Popup 可暂停当前网站、暂停全部、暂停今天、恢复、开始连续学习、打开设置和统计；
- 设置页可管理网站启用状态、兼容状态及基础网页的加载/滚动触发；
- 保存后无需重启扩展，后续学习行为立即使用新值。

**实际**

- 消息层有 `SITE_DISABLE` 与 `PAUSE_ALL`，但当前 Popup 没有“暂停当前网站”和“暂停全部”按钮；
- `PopupDisplayState` 推导了覆盖方式和视频控制能力，UI 只展示兼容等级，未直接显示二者；
- Popup 摘要展示“今日完成题目/长期复习总数/待复习”，第一项却标成“本次学习”，也没有计划要求的今日复习词数、今日新词数或正在学习词数；
- 设置页中的 `pageLoadTrigger` 和 `scrollTrigger` 复选框是 `disabled`，网站启用状态和兼容模式也是只读；
- 没有“复习优先”开关和“连续跳过自动降频”开关，只有不可关闭的优先级与可编辑分钟数组；
- 冷却设置由 background 每次读取，能热更新；但每日新词上限和拼写题开关在内容脚本启动时只读取一次，保存后需要刷新页面才能影响该标签页；词库、水平和长视频设置完全未消费。

**影响**

- 多项已关闭 issue 的验收条件只有底层 API，没有用户可达路径；
- 设置页面存在“保存成功但行为不变”的误导；
- 用户不能完整控制当前网站和基础网页触发频率。

**完成定义**

- Popup 的状态、摘要和所有暂停操作可达、可恢复，并有失败反馈；
- 设置页能修改站点启用状态及基础网页两类触发；
- 删除无效设置，或让所有保留设置在后续自然触发中实时生效；
- 用 Popup/Options 到 background/content 的端到端组件或浏览器测试证明行为，而不只测试消息函数存在。

**已有追踪**：issue #16 已覆盖“设置页手动添加网站”“Popup 顶部加入当前网站”和“基础网页模式主动开始学习”。本 GAP 的新票应排除这些内容，只处理暂停控制、站点/触发编辑、摘要口径和设置热更新。

### GAP-05 本地数据的备份、清除、迁移和并发写入仍不安全

**计划/规格期望**

- 导出包含可恢复的全部本地学习数据；导入先完整校验，错误文件不覆盖现有数据；
- 清除学习进度后统计同步清除；数据库升级不丢数据；
- 同一单词只有一张学习卡，多标签下全局冷却和记录一致。

**实际**

- `sessionLogs` 是跳过数与连续学习统计的唯一来源，但导出结构只含 cards/reviewLogs/words/decks，导入、清除学习进度和清除全部数据也都遗漏 `sessionLogs`；清除后旧统计仍可能出现；
- 导入只验证顶层对象和数组，不逐条校验学习卡、复习日志、单词、词库、站点状态及数值范围；
- 导入先覆盖 `chrome.storage.local`，再清空多个 IDB 仓库，之后逐条并行写入。写入中途失败会留下部分新数据，并已破坏原数据，不满足“错误文件不会覆盖现有数据”；
- 学习卡 `byWordId` 索引不是唯一索引，`getByWordId` 后再 `save` 不是原子操作；两个标签页可竞态创建同一单词的多张学习卡；
- 更新学习卡和写复习日志不在同一 IDB 事务内，任一步失败可能使调度状态与历史不一致；
- `chrome.storage.local` 以一个大对象执行异步 read-modify-write，并发冷却/站点/设置消息存在丢更新风险；
- 长期开启的数据库连接没有 `versionchange` 关闭处理，升级迁移可能被旧标签页阻塞。

**影响**

- 备份不能完整恢复统计；清除不能真正清除；
- 非法或写入失败的导入可能造成不可逆数据损坏；
- 多标签使用可能破坏“唯一学习卡”和全局冷却不变量。

**完成定义**

- 为现有导出格式设计向后兼容的新版本，包含 session logs 与所需源数据；
- 深度校验后，在单一 IDB 事务或安全的 staging/swap 流程中导入，失败保持旧状态不变；
- 清除学习进度与全部数据分别定义并测试所有受影响仓库和统计结果；
- 数据库层强制每个 word 只有一张学习卡，并让提交的卡片更新与复习日志原子提交；
- 串行化共享状态变更，覆盖多标签并发测试；连接响应 `versionchange`，保证升级可完成。

### GAP-06 评分纠正和自动评分输入没有从 UI 贯通到调度

**计划/规格期望**

- 自动评分参考正误、耗时、选项修改、历史正确率和近期答错；
- 用户可在反馈中选择“其实是蒙的”或“这个太简单”，并重新计算本次调度；
- “近期答错”只影响近期，不是永久标签。

**实际**

- `LearningService.correctRating()`、`correct-rating` 动作类型和控制器分支已存在，但 `OverlayApp` 没有任何“蒙的/太简单”按钮，也不会发出 `correct-rating`；
- UI 不统计选项修改次数，提交动作没有传 `answerChanges`，学习服务实际收到默认 0；
- `autoRate()` 用 `recentLogs.some(!isCorrect)` 判断近期错误；`recentLogs` 实际是该卡全部历史。只要历史上答错过一次，之后所有正确答案都会一直被评为 Hard，直到没有任何错误历史这一不可能条件；
- 原计划要求单题“我认识，换一个”立即换下一个候选词；当前单题动作创建自报认识词后直接结束交互。连续模式才会换下一个；
- 原计划要求连续学习中接受新词后至少隔三题再测试；当前短期规则固定十分钟，且会话集合排除已展示词，没有“三题间隔”路径。

**影响**

- 用户纠正评分功能在领域层存在但不可用；
- 自动评分长期偏 Hard，FSRS 间隔和“延迟复习正确率”会被污染；
- 两项旧计划交互尚需明确是删除还是实现，不能保持含糊。

**完成定义**

- 反馈 UI 到调度器完成两种评分纠正闭环，且不新增复习日志、不重复推进 FSRS；
- UI 正确记录答案修改次数；“近期”采用明确时间窗或最近 N 次记录；
- 用行为测试证明一次历史错误不会永久锁定 Hard；
- 产品确认单题换词和连续三题间隔规则，并同步更新计划/规格或实现。

### GAP-07 Beta 核心观察指标还不能可靠验证产品假设

**计划/规格期望**

- 本地观察持续启用、每日自然完成题量、跳过率及变化、主动暂停频率、连续学习使用率、延迟复习正确率；
- 导入、清除、评分纠正后统计仍准确。

**实际**

- 已有完成题、正确题、跳过次数、新词、连续会话、卡片状态和周答题对比；
- 没有站点/插件启用历史，无法衡量“持续启用”；
- 没有暂停操作日志，无法衡量主动暂停频率；
- 只有跳过次数，没有跳过率或趋势展示；
- `delayedReviewAccuracy` 按“当前处于长期阶段的卡片”筛选其全部历史日志，会把进入长期阶段前的首次短期测试也计入，并不能严格代表延迟复习；
- session logs 未纳入导出/导入/清除，进一步破坏统计可恢复性。

**影响**

- 即使分发给测试用户，也无法完整回答计划第二节提出的核心验证问题；
- 指标口径与数据生命周期不一致，可能得到错误结论。

**完成定义**

- 只增加核心假设必需的本地事件和派生指标，不建设后台；
- 明确定义启用、暂停、自然触发、跳过率、连续学习和延迟复习的口径；
- 复习日志记录提交时的学习阶段或等价证据，使延迟复习可准确筛选；
- 所有指标随备份、导入、清除和日期变化保持一致，并在 Popup/统计页使用正确标签。

### GAP-08 主视频识别、触发身份与自定义网站能力需要真实场景加固

**计划/规格期望**

- 主视频需考虑视口可见性、面积、中心距离、播放状态，并过滤广告、预览、背景和小播放器；
- 只有相对于上一内容身份的新视频才触发，布局变化不触发；返回以前看过的视频仍应按当前内容变化处理；
- 自定义站点可在异步视频出现后更新兼容能力；定位不可靠时全页兜底；
- Bilibili 横/竖屏、YouTube/Shorts/直播/广告/SPA、全屏、小窗和多标签通过真实浏览器验收。

**实际**

- 三类视频检测器的 `isVisibleAndMeaningful()` 只检查尺寸和 CSS 可见性，没有检查与视口的相交面积，也没有中心距离评分；页面下方的大视频可能胜过当前视口内播放器；
- `ContentController` 使用永久增长的 `handledIdentities` Set；用户 A → B → A 时，第二次 A 永远不会再成为自然触发点，即使全局冷却已结束；
- Bilibili identity 只含 BV 号，不含分 P 身份；分 P 行为没有明确测试结论；
- 自定义站点只在 bootstrap 时执行一次能力检测。异步视频尚未出现时会固定启动 BasicWebAdapter，本次页面会话不会动态升级；issue #11 的实施评论已把这一点列为遗留；
- 自定义站点被代码禁止使用“完整适配”等级，只能通用视频或基础网页；这与 issue #1 中按能力显示四种状态的描述存在差距；
- 官方适配器在找不到稳定容器时返回一次性 DOMRect，但仍标记 `video-region`；DOMRect 不会随布局变化重新计算，未实现“不可靠则全页”的明确决策；
- 现有 adapter 测试均为 jsdom 构造 DOM；没有 Playwright 扩展 E2E 或仓库内真实站点验收矩阵。

**影响**

- 在复杂页面可能遮错视频、漏触发或永不再触发；
- 自定义网站最关键的“加入后识别视频”可能因加载时序降级；
- 当前测试通过不能替代真实站点兼容性证据。

**完成定义**

- 抽出一个共享、可测试的主视频选择策略，真正计算视口相交与中心/播放权重；
- 用“上一个已处理 identity + 可选定时触发状态”替代永久黑名单，并定义 Bilibili 分 P；
- 自定义站点在页面会话中可从基础网页动态升级到通用/完整能力，权限和 Popup 状态同步；
- 不稳定定位明确降级为全页；
- 为关键浏览器场景建立可重复的 fixture/E2E，加上真实 Chrome/Edge 人工验收记录。

## 六、P1：真实 Beta 前强烈建议完成

### GAP-09 故障恢复仍有用户惩罚和不可恢复分支

- 遮罩宿主被移除或 UI 触发 `recover` 时，控制器把它当作 `skipped`，会增加连续跳过次数并延长冷却；内部错误不应被统计为用户跳过；
- `restore()` 吞掉 `video.play()` 拒绝，调用方无法显示计划要求的“返回视频”恢复动作；
- IndexedDB 打开失败时，统计页显示错误，数据操作返回错误，但没有真正的“导出可读数据后重建数据库”流程；issue #13 对“重建路径”的完成声明高于当前代码能力；
- 数据库/提交失败的会话没有结构化错误记录，Beta 难以区分适配问题、权限问题和数据问题。

完成时应把“用户跳过”“用户结束学习”“内部恢复”分成不同结果，并提供可验证的一次性恢复/重建路径。

### GAP-10 Beta 验收、工程质量门禁与分发体验不完整

- 没有 Playwright 配置、`tests/e2e`、HTML fixture 目录或 GitHub Actions；
- `lint` 脚本不可运行，Prettier、Git hooks、统一日志/错误类型也未建立；
- 完整 `npm audit` 报告开发工具链 14 项漏洞。它们不在打包后的运行时依赖中，但会影响开发/CI 环境，升级需单独验证 WXT/Vite/Vitest 的大版本兼容；
- `dev` 脚本实际执行 development build，而不是计划中的热更新开发模式；
- manifest 没有扩展图标；仓库没有 README、Beta 安装说明、版本化人工验收清单或 Beta 反馈入口；
- issue #13 只记录“基本人工验收通过”，没有 Chrome/Edge 版本、测试 URL、步骤和逐项结果，无法回归。

完成时应建立最小 CI（lint/typecheck/test/build）、可重复浏览器验收、两浏览器安装包与说明、扩展图标及反馈入口；不需要引入与 Beta 核心链路无关的发布平台。

## 七、P2：可在 P0/P1 后优化

### GAP-11 内容脚本和页面观察性能尚无量化保证

- MutationObserver 观察整个 `document.body` 子树，调度器只合并同一微任务内的回调；持续 DOM 更新时仍可能频繁扫描全部 `<video>`；
- 内容脚本约 248 KB，所有获权页面启动时加载完整学习与 UI 代码；
- 671 KB 吉祥物占构建产物的一半以上；
- 没有 CPU、内存、观察器频率或页面隐藏前后的性能基线。

建议先在真实 Bilibili/YouTube/Shorts 上采样，再针对证据做 rAF/节流、范围缩小、按需加载和图片压缩，避免无依据优化。
